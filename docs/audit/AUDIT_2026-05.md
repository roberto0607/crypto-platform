# TRADR — Read-Only Audit (2026-05-17)

Audit scope: full repo, read-only. No code changed. Baseline commit `92a614e` on `main`.

---

## 1. Executive Summary

TRADR is a ~70K-LOC TypeScript monorepo built solo, and the backend is genuinely strong: a real price-time-priority matching engine with documented two-level locking and deadlock avoidance (`matchingEngine.ts:1-62`), strict transactional integrity, `Decimal` math throughout the fill pipeline, an outbox pattern for event delivery, and a coherent ELO/tier/match system. That core is interview-grade work and well above typical portfolio projects. But the project has sprawled — it spans an order book, an ML signal engine, derivatives/gamma data, "cycle intelligence," and liquidation estimators — and the frontend has rotted around the edges: **8 page files totaling ~5,400 LOC are completely unrouted dead code**, including the 1,132-LOC `ReplayPage.tsx` that CLAUDE.md calls the "priority competitive feature." The web app has exactly one test file and 7 of its 8 tests fail; the local dev database no longer exists (the port-5433 container was repurposed for a different project). The last three weeks of git history are documentation-only — the project stalled mid-rollout.

**Honest grades:** Backend engineering craft **B+ / A-** (the matching engine in isolation is A material). Frontend **C** (works, but desktop-only, poor a11y, silent failures, half of it dead). **Overall as a resume artifact: B-** — held back by sprawl and the dead frontend, not by the engineering. **As a startup: D+** — paper trading with no monetization path, no telemetry, no onboarding, no retention loop, no mobile, and scope spread thin instead of a focused wedge.

## 3. Top 10 Gaps Ranked by Impact

### 1. ~5,400 LOC of orphaned, unrouted frontend pages
**What:** 8 page files have zero references anywhere in the app — `App.tsx` routes their paths to redirects instead (`App.tsx:213-221`): `ReplayPage.tsx` (1,132 LOC), `DashboardPage.tsx` (1,017), `PortfolioPage.tsx` (926), `CompetitionsPage.tsx` (621), `CompetitionDetailPage.tsx`, `OrderHistoryPage.tsx`, `PositionsPage.tsx`, `TriggersPage.tsx`, plus `AdminPage.tsx` (a `<div>Admin — TODO</div>` placeholder, `AdminPage.tsx:2`). `WalletsPage.tsx` is also unrouted.
**Why it matters:** Anyone reviewing this repo — recruiter or engineer — sees a codebase where ~18% of the frontend is dead. It reads as abandoned WIP. It also bloats every grep, refactor, and typecheck.
**Effort:** S (delete) or M (re-route what's still wanted). **Priority:** Resume — high. Startup — high (the replay feature is a real product asset being thrown away).

### 2. The "priority competitive feature" (Post-Match Replay) is built but unreachable
**What:** CLAUDE.md Stage 6 calls Post-Match Replay the priority competitive feature. The backend exists (`replayRoutes` registered at `app.ts:225`), `ReplayPage.tsx` (1,132 LOC) and `ReplayChart.tsx` exist, `api/endpoints/replay.ts` has 7 endpoints, SSE `replay.tick` is wired. But `/replay` redirects to `/trade` (`App.tsx:218`) and nothing imports `ReplayPage`. Two UI strings still tell users to "Go to Replay" (`BotPage.tsx:324`) and "Start one from the Replay page" (`ReplayChart.tsx:279`) — pointing at a page that doesn't route.
**Why it matters:** This is the single most demo-worthy competitive feature and it's invisible. Months of work producing zero user-facing value.
**Effort:** M (re-route + nav link + smoke test). **Priority:** Resume — high. Startup — high.

### 3. Local dev environment has rotted
**What:** The TRADR dev DB (`cp:cp@localhost:5433/cp`) no longer exists. The container on port 5433 (`cp_postgres`) is now serving a different project (`POSTGRES_DB=ai_trading_agent`, user `postgres`). The `cp` role and `cp` database are gone. Consequently the DB sanity checks requested in this audit (positions with `match_id IS NULL`, unscoped wallets, `schema_migrations` version) **could not be run** and 9 API test files fail.
**Why it matters:** You cannot currently run the full test suite or demo the app locally without rebuilding the database. "Clone and run" is broken.
**Effort:** S (recreate DB + run migrations) — but it should be a documented one-command bootstrap. **Priority:** Resume — high (a broken local setup torpedoes a take-home/walkthrough). Startup — medium.

### 4. `cancelActiveMatch` anti-gameability hole
**What:** `cancelActiveMatch` allows cancelling an ACTIVE match only if it has zero trades, checked via `match.challenger_trades_count + opponent_trades_count` (`matchService.ts:463-466`). But those columns are **only written by `completeMatch`/`forfeitMatch`** (`matchService.ts:235-236, 335-336`) — they are never incremented during a live match. So during an ACTIVE match the counters are always 0, and a player sitting on a losing open position can call `cancelActiveMatch` instead of forfeiting — escaping the loss, the ELO hit, and the `closeMatchScopedPositions` book-the-loss mechanism entirely.
**Why it matters:** It directly defeats the anti-gameability guarantee that commit `e54b14a` was written to enforce. In a competitive ranking product this is exploitable.
**Effort:** S (count live trades from `orders WHERE match_id = ... AND status = 'FILLED'`, the same query `calculatePlayerStats` already uses). **Priority:** Resume — medium (good bug to have caught and fixed). Startup — high.

### 5. Web app has effectively no test coverage and its one test is broken
**What:** `apps/web/src` has a single test file; 7 of its 8 tests fail because the component DOM changed and the test wasn't updated (`ordersTab.test.tsx:177`). 29K LOC of frontend, ~0 effective coverage.
**Why it matters:** A failing test in the repo is worse than no test — it signals nobody runs the suite. Frontend regressions (the price hero hardcoded to "up", §6) ship silently.
**Effort:** M. **Priority:** Resume — high (interviewers look). Startup — medium.

### 6. User-initiated actions fail silently
**What:** Forfeit, accept, and TP/SL edits swallow errors with empty catches: `LiveMatchView.tsx:618` (forfeit), `ArenaPage.tsx:295` (accept), `ArenaPage.tsx:304` (forfeit), `UnifiedOrderPanel.tsx:335` (TP/SL edit). User clicks, nothing happens, no error, no feedback.
**Why it matters:** A forfeit that silently fails is a trust-breaking bug — the user thinks they quit, the match keeps running, ELO is still at stake.
**Effort:** S. **Priority:** Resume — medium. Startup — high.

### 7. Scope sprawl with large unmaintained surfaces
**What:** Beyond the trading core, the repo carries `api/market` (24 files, 5,984 LOC), `api/strategy` (24 files, 4,343 LOC) — ML signals, regime classifier, options gamma, liquidation estimator, macro correlation, cycle intelligence. None of `api/strategy` or `api/market` has tests. Recent git history (`e4de89a`..`0e922a9`) shows a long debugging slog through liquidation/COT/OI features.
**Why it matters:** For a resume, breadth-without-depth reads as unfocused. For a startup, it's effort spent on five half-products instead of one sharp one. These modules are also the most likely to silently break.
**Effort:** L (decide what to cut). **Priority:** Resume — high (focus the narrative). Startup — high (focus the product).

### 8. 234 `console.log` calls in `apps/api/src`
**What:** 234 `console.log` lines in API source (non-test), concentrated in `api/market` (13 files) and `api/scripts` (8 files — acceptable for CLIs). Plus `console.log` debug traces in `CandlestickChart.tsx` (lines ~307, 757, 772-805 — an entire `[liq]` debug trace). The project has a structured `logger` (`observability/logContext.ts`) that this bypasses.
**Why it matters:** Console noise in request-path code is unprofessional in a review and unstructured in production. Recent commits literally include "strip ... diagnostic logs" (`58c8b2f`) and "debug: log ..." — the cleanup was incomplete.
**Effort:** S. **Priority:** Resume — medium. Startup — low.

### 9. Zero telemetry / product analytics
**What:** There is operational observability (Prometheus metrics in `metrics.ts`, audit log) but no product analytics — no event tracking for signups, match starts, retention, funnel, feature usage. `App.tsx` init fetches status/wallets but emits no analytics events.
**Why it matters:** As a startup this is disqualifying — you cannot tell if anyone uses anything, cannot measure activation or retention, cannot make a data-driven pitch.
**Effort:** M. **Priority:** Startup — high. Resume — low.

### 10. Dead code in the match service
**What:** `forceCloseOpenPositions` (`matchService.ts:793-868`) is explicitly marked DEPRECATED and "Safe to remove" — never called. `checkMatchMinimumRequirements` (`matchService.ts:276-290`) is never called and queries the deprecated `match_positions` table — if it were wired in it would always return 0 trades. `MIN_MATCH_TRADES` (`matchService.ts:59`) exists but is never enforced — `completeMatch` ranks matches regardless of trade count.
**Why it matters:** Dead code with stale table references is a trap for the next reader (and for you in 3 months). The unenforced minimum-trades rule means a 0-trade match still produces an ELO result.
**Effort:** S. **Priority:** Resume — medium. Startup — low.

## 5. Architecture Red Flags

**What scares a senior engineer:**
- The dead-frontend problem (§3.1) — it reads as "the author lost the thread." Routes redirecting to `/trade` instead of pages being deleted suggests an unfinished cleanup that was never finished.
- One web test, failing. Implies the suite is not run; CI (if any — `.github/` exists, not audited) is not gating.
- 9 test files that *error* rather than *skip* when the DB is absent — the test harness doesn't degrade cleanly. "260 green" is a soft claim.
- 234 `console.log` in API source despite a structured logger existing — inconsistency between intent and practice.
- The anti-gameability bug (§3.4) — the *mechanism* (`closeMatchScopedPositions`) is well-built, but a sibling code path (`cancelActiveMatch`) bypasses it. This is the classic "secure the front door, leave the side door open."

**What a senior engineer would respect:** The matching engine's locking design (`matchingEngine.ts:1-62`) is documented at a level most production systems aren't — lock ordering, deadlock invariant, and an honest "future: narrowing the pair lock" analysis. The single-transaction fill pipeline with outbox events (`phase6OrderService.ts:168-434`), idempotency-key handling with race recovery (`phase6OrderService.ts:436-449`), and the maker-scope bug fix comment (`phase6OrderService.ts:241-244`) show real distributed-systems care. ELO resolution is idempotent and lock-ordered (`eloService.ts:resolveMatchElo`). This person can engineer.

**What scares a YC partner:**
- It's paper trading with no monetization path and no plan for one — §9.
- No analytics — you cannot answer "do people come back?" — §3.9.
- Scope sprawl: an order book *and* an ML signal engine *and* derivatives data *and* cycle forecasting. A YC partner asks "what is the one thing?" and the repo has no answer.
- 3 weeks of no feature commits, last 5 commits are docs/postmortem/"V4 deferral" — momentum has stalled.
- No onboarding, no mobile (§6) — the activation funnel doesn't exist.

## 6. UX / Frontend Gaps

(From a focused read of `UnifiedOrderPanel.tsx`, `LiveMatchView.tsx`, `TradingPage.tsx`, `ArenaPage.tsx`, `CandlestickChart.tsx`.)

- **No responsive design anywhere.** Fixed two-column grids (`LiveMatchView.tsx:645` `1fr 300px`; `TradingPage.tsx:130-137` with `height: calc(100vh - 126px)`; `ArenaPage.tsx:42` `repeat(5,1fr)`), fixed pixel font sizes, zero media queries, zero Tailwind responsive prefixes in any of the five files. The app is unusable on a phone or tablet.
- **Accessibility is poor.** Interactive controls are `<div onClick>`/`<span onClick>` — not keyboard operable: direction/order-type/leverage toggles (`UnifiedOrderPanel.tsx:360-407`), "MAX" (`:431`), TP/SL edit controls (`:547-571`), asset tabs (`TradingPage.tsx:759-768`), arena tabs (`ArenaPage.tsx:392-394`). Labels are not associated to inputs via `htmlFor`/`id`. No focus management, no `role="dialog"`/focus trap on modals. **Exceptions (good):** `ArenaPage`'s challenge form and match-history `<table>` are properly semantic; `CandlestickChart` timeframe buttons are real `<button>`s; `UnifiedOrderPanel`'s submit/close are real buttons.
- **Loading states:** mostly text-only ("PLACING...", "CANCELLING...") — minimal but present. `LiveMatchView` has no loading indicator at all for position/match fetch.
- **Error states:** `ArenaPage` is the high-water mark — a real tri-state season card (loading / "SERVER OFFLINE" / data, `ArenaPage.tsx:340-353`) and specific error-code branching with a "CANCEL STUCK MATCH" recovery action (`:258-265, 491-500`). Elsewhere errors are swallowed (§3.6). `CandlestickChart` candle-fetch failure renders a blank black chart with no message (`CandlestickChart.tsx:563-565`).
- **Empty states:** present where it matters — "NO PAIRS AVAILABLE", order book "WAITING FOR BOOK DATA..." (`TradingPage.tsx:564-569`), "NO MATCHES YET" (`ArenaPage.tsx:510`).
- **Visible bug:** the price hero is hardcoded to the `up` (green) CSS class regardless of actual price direction — `TradingPage.tsx:774`.
- **Genuinely good polish:** order-book math deliberately uses raw (un-formatted) prices to avoid `parseFloat` comma bugs, with an explaining comment (`TradingPage.tsx:538-557`); chart `fitContent()` runs only on first load so indicator toggles don't yank the viewport (`CandlestickChart.tsx:558-562`); `isMounted` guards against unmounted-component `setState` (`LiveMatchView.tsx:522-527`); idempotency key on close orders (`UnifiedOrderPanel.tsx:311`).

## 7. Security & Data Integrity

- **Secrets:** `.env` is gitignored; the tracked `apps/api/.env.development` contains only dev bypass flags (`DISABLE_RATE_LIMIT`, inflated login attempt caps, etc.) — no real secrets. CLAUDE.md prints a production `DATABASE_URL` with a masked password and real host/port — that file is committed; the host:port is exposed but the password is masked. Worth scrubbing.
- **Auth:** Argon2 + JWT access + HttpOnly refresh-cookie rotation with reuse detection and family revocation — a strong design. Known limitation already documented: multi-tab refresh race (CLAUDE.md "Known Issues").
- **Rate limiting:** present (`app.ts`, `authRoutes.ts`, `tradingRoutes.ts`) with a `DISABLE_RATE_LIMIT` escape hatch gated to non-production.
- **Injection:** No ORM, all raw SQL via `pg` — spot-checked `matchingEngine.ts`, `matchService.ts`, `eloService.ts`, `phase6OrderService.ts`: every query is parameterized (`$1..$n`). The one place SQL is string-built is the dynamic `VALUES` list in `createMatch` (`matchService.ts:104`) and the `compFilter` fragment in `matchingEngine.ts:242-256` — both build only placeholder text, never interpolate user data. No injection risk found in the audited files.
- **Demo account isolation:** CLAUDE.md flags the `demo@demo.local` account was inserted directly into the DB outside the auth flow, and that `orders.competition_id` must never be set to a match UUID (FK points to `competitions`, not `matches`). These are known traps, documented, not verified live (DB unavailable).
- **Data integrity — could not verify:** the requested DB sanity checks (positions with `match_id IS NULL`, unscoped wallets, current `schema_migrations` version) **were not run** — the local dev DB does not exist (§3.3). From source: latest migration file is `066_position_match_scope.sql`; migration 066 is clean, idempotent (guarded `pg_constraint` lookups, `IF NOT EXISTS`), and explicitly does not backfill — all existing rows keep `match_id IS NULL` (free play), which is by design. The match-scoping unique-index strategy (COALESCE-nil-UUID) is sound.
- **Anti-gameability gap:** see §3.4 — `cancelActiveMatch` lets a player escape a losing match.

## 8. What's Actually Resume-Standout

Three things to point an interviewer at:

1. **The matching engine** — `apps/api/src/trading/matchingEngine.ts`. A real price-time-priority limit order book with a documented two-level locking strategy (pair lock → wallet locks in sorted UUID order), an explicit deadlock-avoidance invariant, cursor-based keyset pagination over the resting book to cap memory, and an honest written analysis of how you'd narrow the pair lock for throughput and why you haven't. The hard part was correctness under concurrency — and the lock-ordering proof and `FOR UPDATE` double-read in `cancelOrderInternal` (`:533-540`) show you reasoned about the races, not just hoped.

2. **The transactional fill pipeline** — `apps/api/src/trading/phase6OrderService.ts`. Single transaction spanning matching + position updates + journal + outbox + portfolio snapshot, with SSE events deferred until after commit, idempotency-key dedup *with* a post-commit race-recovery path (`:436-449`), and the maker-scope bug fix (`:241-288`) where you correctly took the maker's position scope from *their* original order rather than the incoming taker's. That's a subtle correctness bug most people would ship.

3. **Match-scoping migration 066 + the ELO/tier system** — `migrations/066_position_match_scope.sql` is a model of how to write a migration: context, what-it-does, what-it-doesn't, paired code changes, reversibility. Pair it with `eloService.ts:resolveMatchElo` — idempotent, row-locked, with streak multipliers, asymmetric tier-based deltas, and promotion/demotion all resolved in one transaction. (Honest caveat for the interview: call it a "tier-based competitive rating system," not "ELO" — it deliberately does *not* use the classic K-factor expected-score formula.)

## 9. What's Startup-Missing

- **Monetization:** none, and no path. Paper trading has no inherent revenue. Viable models (subscriptions for advanced indicators, entry fees / prize pools for competitions, a "pro" tier) exist but nothing in the codebase moves toward any of them.
- **Telemetry / analytics:** none (§3.9). No activation, retention, or funnel measurement. You can't pitch growth you can't see.
- **Onboarding:** no first-run flow, no tutorial, no guided first trade or first match. New user lands on a dense desktop trading screen.
- **Retention loop:** matches + ELO + streaks are the *bones* of a retention loop, but there's no notification/email re-engagement surface driving users back to it. `api/email` and `api/notifications` exist (small) but there's no "your rival challenged you" / "your streak is at risk" push.
- **Mobile:** zero responsive design (§6). The entire addressable consumer-trading audience is mobile-first; this app is desktop-only.
- **Growth surface:** no shareable artifact. The replay feature (§3.2) is the obvious viral unit — "watch how I won this match" — and it's unrouted. No public profiles, no shareable match results, no leaderboard embed.
- **Focus:** the product is an order book *and* a signal engine *and* a derivatives terminal *and* a cycle forecaster. A startup needs one wedge. The competitive 1v1 match with replay is the most differentiated, most viral, most defensible — everything else is a distraction from it.
